# Monetization Decision Matrix

Qlass should remain open source and self-hostable while building revenue around convenience, reliability, and specialized workflows.

| Path | Pros | Cons | Recommendation |
| --- | --- | --- | --- |
| Hosted teacher/small-school tier | Most passive, easy to explain, preserves open-source core. | Requires uptime, backups, billing, privacy posture, and support. | Attack first. |
| Paid plugins/connectors | Strong fit for migrations and SIS/LMS interoperability. | Connectors are brittle and support-heavy. | Build second after hosted alpha. |
| Support/services | Realistic for schools that need help deploying. | Less passive and can become consulting. | Use selectively to learn requirements. |
| Agent packs/templates | Differentiated and scalable once users exist. | Needs a user base and quality controls first. | Build third. |

## Chosen first path

Start with a hosted Qlass alpha for individual teachers and small schools. The free/open-source version remains installable, while paid hosting removes setup pain and creates the clearest path to recurring revenue.

## First pricing hypothesis

- Free: self-hosted, community support.
- Teacher hosted: low monthly price for one teacher, limited storage/classes.
- Small-school hosted: higher monthly price for multiple teachers, backups, admin controls, and support.
- Later add paid connector packs or managed migration services.
